Plan

Flow Scope V3 Refactor

Replace v2 table sprawl with a dimensioned stats model that treats anonymization visibility as a first-class query dimension.

Core tables 4 dimensioned stats tables
Scope model src_visibility and dst_visibility
Aggregation One table per concept, all granularities in-row
Compatibility all/all rows preserve current dashboard semantics

Goal

Introduce anonymization-aware analysis without creating scoped sibling tables for every existing v2 table. The v3 model should keep table count small, make UI filtering straightforward, and avoid overcounting unique address and address-structure metrics.

Store anonymization as visibility, not as internal/external truth. ToS low bits identify which side was anonymized. Known literal UO ranges can be layered on later as separate address labels.

Decisions

  • Use v3 tables beside v2 during migration.
  • V3 replaces v2 after parity and app migration are complete.
  • Do not add separate aggregate tables; use granularity in every v3 table.
  • Use src_visibility and dst_visibility, not a single opaque flow_scope column.
  • Use address_side for source/destination address-set dimensions.
  • Persist all/all rows explicitly for current app behavior.
  • Compute unique counts and address structure from raw sets, never by summing scoped rows.
  • Defer UO-specific CIDR labels.
  • Expose exact visibility pairs only in the first UI slice.

Visibility Semantics

Field Values Meaning
src_visibility all, literal, anonymized Source address visibility for a row.
dst_visibility all, literal, anonymized Destination address visibility for a row.
address_side source, destination Which address set a count or structure row describes.
structure_kind structure, spectrum, dimension Which address-shape output is stored in values_json.

ToS Decode

src_anonymized = (src_tos & 2) != 0
dst_anonymized = (src_tos & 1) != 0

src_visibility = src_anonymized ? 'anonymized' : 'literal'
dst_visibility = dst_anonymized ? 'anonymized' : 'literal'

Current samples show dst_tos is zero. Keep it available for audit, but derive visibility from src_tos low bits.

Tables

traffic_stats_v3

CREATE TABLE traffic_stats_v3 (
  source_id TEXT NOT NULL,
  granularity TEXT NOT NULL CHECK (granularity IN ('5m', '30m', '1h', '1d')),
  bucket_start INTEGER NOT NULL,
  bucket_end INTEGER NOT NULL,
  ip_version INTEGER NOT NULL CHECK (ip_version IN (4, 6)),
  src_visibility TEXT NOT NULL CHECK (src_visibility IN ('all', 'literal', 'anonymized')),
  dst_visibility TEXT NOT NULL CHECK (dst_visibility IN ('all', 'literal', 'anonymized')),
  flows INTEGER NOT NULL,
  flows_tcp INTEGER NOT NULL,
  flows_udp INTEGER NOT NULL,
  flows_icmp INTEGER NOT NULL,
  flows_other INTEGER NOT NULL,
  packets INTEGER NOT NULL,
  packets_tcp INTEGER NOT NULL,
  packets_udp INTEGER NOT NULL,
  packets_icmp INTEGER NOT NULL,
  packets_other INTEGER NOT NULL,
  bytes INTEGER NOT NULL,
  bytes_tcp INTEGER NOT NULL,
  bytes_udp INTEGER NOT NULL,
  bytes_icmp INTEGER NOT NULL,
  bytes_other INTEGER NOT NULL,
  processed_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (
    source_id, granularity, bucket_start, ip_version,
    src_visibility, dst_visibility
  )
) WITHOUT ROWID;

Pipeline Plan

  1. Add a shared visibility classifier used by CSV and native nfcapd paths.
  2. Add v3 table init and insert helpers.
  3. Extend 5m bucket state to maintain all/all and exact visibility pairs.
  4. For nfcapd traffic, group with nfdump -a -A proto,srctos,dsttos.
  5. For nfcapd address sets, group with srcip,dstip,srctos,dsttos.
  6. For CSV, decode visibility from normalized src_tos.
  7. Roll up 30m, 1h, and 1d from raw sets plus additive traffic counters.
  8. Run address-structure calculations per exact scope and for all/all.
Additive metrics can sum. Unique address counts, protocol sets, and address structures require set union before writing each granularity.

App Plan

  1. Keep existing v2 routes until v3 parity is verified.
  2. Add v3 query helpers that always filter granularity and visibility.
  3. Default dashboard queries use src_visibility='all' and dst_visibility='all'.
  4. Add controls for exact visibility pairs only.
  5. Migrate charts route-by-route: traffic, protocol, address count, address structure.
  6. Delete v2 readers and tables only after v3 tables cover all current UI views.

Validation

  • For every bucket, all/all traffic equals the sum of the four exact visibility pairs.
  • For protocols, all/all protocol set equals union of exact visibility-pair protocol sets.
  • For address counts, all/all counts are computed from raw address sets, not summed.
  • For address structure, all/all is computed from raw address sets, not combined JSON.
  • Audit dst_tos != 0, anonymized-looking addresses with clear bits, and literal known ranges with set bits.
  • Compare v2 and v3 all/all rows before switching app defaults.

Rollout

  1. Schema and insert helpers.
  2. CSV path v3 output with focused tests.
  3. Native nfcapd path v3 output with nfdump parser tests.
  4. Aggregate rollups for all granularities.
  5. Address-structure v3 writer and parity tests.
  6. App read path behind v3 query helpers.
  7. Backfill/reprocess selected UOregon days and verify parity.

Resolved Questions

  • V3 will replace v2 after parity is proven.
  • UO-specific address labels are out of scope for this refactor.
  • The first UI slice exposes exact visibility pairs only.
  • Address-structure processing should do whatever work is needed; current timeout limits are acceptable.
